## 작업 요약

3대 맥(M4 Studio 128GB, M4 Air 32GB, M1 Pro 16GB)의 컴퓨팅 자원 최대 활용을 위해 모바일 최적화 프로젝트 대시보드를 기획하고, 각 머신별 오늘의 작업을 배분하여 GitHub에 푸시했다. 아키텍트 검토를 통해 초기 Gist 기반 설계의 보안/API 제한 문제를 발견하고 NAS+Studio 중앙 수집 방식으로 전환했다.

## 상세 작업 내역

### 1. 3대 머신 상태 조사

SSH로 M4 Air, M1 Pro 연결 확인. 각 머신의 CPU/MEM/프로세스/launchd 상태를 수집했다. Air는 node 프로세스만 잔류(유휴), M1 Pro는 Claude 앱+Ollama+salpim sentinel만 동작(유휴). Studio는 Ollama(42% MEM) + PostgreSQL 가동 중.

### 2. 프로젝트 대시보드 아키텍처 설계

모바일에서 폰으로 확인 가능한 웹 대시보드를 설계. 초기안(GitHub Gist + Static SPA)에서 아키텍트 검토 후 NAS+Studio 중앙 수집 방식으로 전환. 기존 `machine-status.schema.json` 스키마를 재활용하고, 브라우저에서 API 콜 0회로 동작하는 구조로 확정.

### 3. 머신별 태스크 배분 + 핸드오프

- M4 Studio: ai-news-pipeline Phase 2 + content-marketing-team 통합
- M4 Air: project-tracker 웹 대시보드 빌드 (4단계 상세 가이드)
- M1 Pro: salpim-web + urban-jungle + copper-briefing 점검
- 핸드오프 문서 작성하여 출근 후 auto-checkin으로 즉시 로드 가능하게 함

### 4. 아키텍트 검토로 보안/성능 문제 수정

- 공개 Gist에 머신 정보 노출 → NAS 비공개 경로로 전환
- 클라이언트 JS 토큰 노출 → Studio 서버사이드로 이동
- GitHub API 60회/시간 제한 초과 → 사전 수집 JSON 방식으로 0회
- CDN 의존 → 커밋된 CSS로 전환

## 핵심 결정

- **기존 project-tracker 레포 대신 homelab-orchestration에 대시보드 추가**: 이미 tasks/handoffs/schemas 인프라가 구축돼 있고, NAS-First 원칙과 일관성 유지
- **Gist → NAS+Studio 수집 방식**: 보안(토큰 비노출), 성능(API 콜 0회), 오프라인(로컬 폴백) 모두 해결
- **빌드 없는 Vanilla SPA**: 외부 의존 제로, CDN 없음, homelab-orchestration 철학 준수
- **M4 Air를 대시보드 빌드 전담**: 32GB로 프론트엔드 개발에 적합, Studio는 AI 작업에 집중
